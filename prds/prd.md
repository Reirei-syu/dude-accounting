# Dude Accounting PRD

## 1. 文档目的
本文档用于重新定义 Dude Accounting 的产品定位、合规边界与分阶段改造目标，确保项目从“本地桌面记账应用雏形”收敛为“代理记账企业为委托单位记账使用的单机版财务软件”。

本轮 PRD 重点解决两类问题：
- 产品定位与当前代码实现、合规要求之间存在偏差。
- 会计软件合规能力缺口已经影响核心架构与后续开发顺序。

相关合规依据见：
- `电子会计档案管理规范`
- `电子凭证会计数据标准应用指南`
- `会计软件基本功能和服务规范`
- `会计信息化工作规范`

## 2. 产品定位
- 产品名称：Dude Accounting
- 产品形态：桌面端单机财务软件
- 使用主体：代理记账企业内部人员
- 服务对象：委托单位
- 委托单位范围：企业、民间非营利组织（`enterprise` / `npo`）
- 明确排除：政府会计、事业单位会计
- 核心场景：一个软件实例中维护多个委托单位账套，完成凭证处理、账套维护、结账、账簿报表、电子凭证处理、电子会计档案导出与备份

## 3. 当前产品真实状态

### 3.1 已实现
- 本地账号登录与权限校验
- 多账套管理、会计期间切换
- 企业/民非账套模板初始化
- 凭证录入、审核、记账、删除、查询
- 会计科目与辅助账基础维护
- 账套设置中的期末损益结转规则维护，可按末级损益科目配置结转目标
- 期初余额录入
- 期末损益结转、期间结账/反结账
- 现金流量项目与匹配规则维护

### 3.2 当前合规硬缺口
- 已记账凭证当前仍支持反记账，不满足“已记账不可逆”的合规方向
- 未建立统一的操作日志、查询与导出能力
- 电子凭证标准化处理链路缺失：接收、验签/验真、解析、去重、留痕、结构化入账
- 电子会计档案导出与归档接口缺失
- 账套备份/恢复尚未打通，菜单入口为占位
- 账簿与报表输出仍为占位模块，不满足最低合规输出能力

### 3.3 已确认的产品决策
- `ledger` 继续表示“一个委托单位的一套账”
- 普通用户不再允许对已记账凭证执行反记账
- 保留管理员“紧急逆转”例外，但必须记录理由、审批标记与操作日志
- 电子档案优先做“标准导出与备份”，暂不做完整内置档案库
- 电子凭证第一阶段优先支持：数电发票、银行电子回单/对账单

## 4. 改造目标

### 4.1 目标一：修正产品边界
- 明确本产品是代理记账企业使用的软件，而不是面向所有单位通用部署的会计平台
- 明确支持范围仅为企业和民非账套
- 将“账簿、报表、备份、电子凭证、电子档案、日志”从未来可选项提升为合规必选项

### 4.2 目标二：完成核心控制整改
- 取消普通用户对已记账凭证的反记账路径
- 建立管理员紧急逆转机制
- 建立操作日志与关键业务过程留痕
- 收紧账套删除前置条件

### 4.3 目标三：搭建电子凭证与电子档案底座
- 新增电子凭证记录、校验、去重、解析、入账关联
- 输出归档所需元数据、结构化入账文件和导出清单
- 区分“系统备份”与“归档导出”

### 4.4 目标四：补齐最低输出能力
- 补齐账簿查询
- 补齐报表输出
- 补齐打印、版式导出、标准数据导出接口

## 5. 分阶段实施范围

### Phase 0：文档与范围校准
- 重写 PRD、PROJECT_SPEC
- 生成单独的合规整改清单
- 明确每一条规范对应的当前差距和实施阶段

### Phase 1：核心账务控制整改
- 普通用户取消 `unbookkeep`
- 管理员紧急逆转
- 操作日志
- 账套删除前置校验

### Phase 2：电子凭证标准化底座
- 电子凭证原件记录
- 验签/验真状态留痕
- 结构化解析结果存储
- 重复入账拦截
- 记账凭证与来源电子凭证关联

### Phase 3：电子档案与备份导出
- 账套备份包生成、校验、恢复
- 电子会计档案导出包生成
- 导出清单、元数据、校验信息、可读性检查记录

### Phase 4：账簿报表与标准输出
- 总账余额表、科目余额表、序时账、辅助账
- 资产负债表、利润表/业务活动表、现金流量表
- 打印、版式导出、标准数据导出

## 6. 关键业务规则

### 6.1 账套与适用范围
- 一个账套对应一个委托单位
- 一个委托单位仅允许选择 `enterprise` 或 `npo`
- 不提供政府会计和事业单位模板

### 6.2 凭证控制
- 已记账凭证默认不可逆、不可删除、不可修改关键字段
- 普通用户仅能对未记账凭证执行审核、反审核、删除等允许动作
- 管理员紧急逆转必须记录理由、审批标记和时间
- 系统必须具备防止电子会计凭证重复入账的能力
- 末级损益科目必须先配置结转目标，未完整配置或目标失效时不得执行期末损益结转

### 6.3 电子凭证处理
- 原始电子凭证需保留来源、类型、校验状态、解析状态和摘要信息
- 解析出的结构化结果需可追溯到原文件
- 入账后需建立“原始电子凭证 -> 记账凭证 -> 归档导出物”的关联链路

### 6.4 电子档案与备份
- 备份包与档案导出包必须分离
- 档案导出包必须保留元数据、校验信息和导出记录
- 备份恢复必须先校验包完整性

### 6.5 输出能力
- 软件应能输出会计凭证、账簿、报表
- 输出方式至少包含：显示、打印、版式导出、标准数据导出

## 7. 非功能要求
- 全部核心数据本地存储
- 所有关键业务操作需要可查询日志
- 重要导出与恢复动作要有结果反馈与失败原因
- 文档统一 UTF-8 中文
- 新增模块优先通过 IPC 和服务层扩展，不把复杂规则下沉到 renderer

## 8. 验收标准

### 8.1 文档
- `prds/prd.md`、`prds/PROJECT_SPEC.md`、`prds/合规整改计划.md` 与当前代码状态一致
- 产品定位、范围边界和分阶段任务明确

### 8.2 核心控制
- 普通用户无法反记账已记账凭证

- 能创建电子凭证记录并执行重复入账拦截

- 能生成电子会计档案导出包并保留导出记录

- `npm test` 通过
