# Dude Accounting PRD

## 1. 文档目的

本文档用于重新定义 Dude Accounting 的产品定位、合规边界与分阶段改造目标，确保项目从“本地桌面记账应用雏形”收敛为“代理记账企业为委托单位记账使用的单机版财务软件”。

协作基线：

- 所有交流统一使用简体中文；如无特殊说明，项目内新增或修改的文案默认使用简体中文。
- 每次完成一次重要修改后，必须先做 review，再执行验证；验证通过后再向用户交付成果。
- 当前打开的 folder 视为项目级最底部目录；除非用户明确要求，新增文件默认放在当前项目目录内。
- 如用户要求打包为单独文件、安装包或其他可交付安装产物，输出目录统一为 `D:\coding\completed\项目名称`。

本轮 PRD 重点解决两类问题：

- 产品定位与当前代码实现、合规要求之间存在偏差。
- 会计软件合规能力缺口已经影响核心架构与后续开发顺序。

相关合规依据见：

- `电子会计档案管理规范`
- `电子凭证会计数据标准应用指南`
- `会计软件基本功能和服务规范`
- `会计信息化工作规范`

## 2. 产品定位

- 产品名称：Dude Accounting
- 产品形态：桌面端单机财务软件
- 使用主体：代理记账企业内部人员
- 服务对象：委托单位
- 委托单位范围：企业、民间非营利组织（`enterprise` / `npo`）
- 明确排除：政府会计、事业单位会计
- 核心场景：一个软件实例中维护多个委托单位账套，完成凭证处理、账套维护、结账、账簿报表、电子凭证处理、电子会计档案导出与备份

## 3. 当前产品真实状态

### 3.1 已实现

- 本地账号登录与权限校验
- 多账套管理、会计期间切换
- 企业/民非账套模板初始化
- 凭证录入、审核、记账、删除、查询
- 会计科目与辅助账基础维护
- 账套设置中的期末损益结转规则维护，可按末级损益科目配置结转目标
- 期初余额录入
- 期末损益结转、期间结账/反结账
- 现金流量项目与匹配规则维护
- 财务报表快照生成、报表查询、删除与另存为导出，支持资产负债表按月份生成、动态报表按跨月区间生成，支持“未记账凭证”口径和按报表类型/会计期间多选筛选已生成报表；导出格式支持 Excel 与 PDF，民非三张报表按官方会民非 01/02/03 表制式显示
- 系统级数据库快照备份包生成、manifest 清单写入、完整性校验与整库恢复
- 新建账套会拦截与既有账套完全同名的名称；备份与归档创建支持自定义保存目录，恢复支持从自选备份包目录发起

### 3.2 当前合规硬缺口

- 已记账凭证当前仍支持反记账，不满足“已记账不可逆”的合规方向
- 未建立统一的操作日志、查询与导出能力
- 电子凭证标准化处理链路缺失：接收、验签/验真、解析、去重、留痕、结构化入账
- 电子会计档案当前已支持导出与包级校验，但结构化入账文件校验和更完整的可读性检查仍未补齐
- 账套备份当前已支持系统级数据库快照包生成、manifest 校验与整库恢复，但仍不支持单账套局部恢复与异地备份策略
- 账簿查询尚未完成；报表输出已具备基础快照生成、查询、删除与 Excel/PDF 另存为导出，但打印、标准数据导出仍未补齐

### 3.3 已确认的产品决策

- `ledger` 继续表示“一个委托单位的一套账”
- 普通用户不再允许对已记账凭证执行反记账
- 保留管理员“紧急逆转”例外，但必须记录理由、审批标记与操作日志
- 电子档案优先做“标准导出与备份”，暂不做完整内置档案库
- 电子凭证第一阶段优先支持：数电发票、银行电子回单/对账单

## 4. 改造目标

### 4.1 目标一：修正产品边界

- 明确本产品是代理记账企业使用的软件，而不是面向所有单位通用部署的会计平台
- 明确支持范围仅为企业和民非账套
- 将“账簿、报表、备份、电子凭证、电子档案、日志”从未来可选项提升为合规必选项

### 4.2 目标二：完成核心控制整改

- 取消普通用户对已记账凭证的反记账路径
- 建立管理员紧急逆转机制
- 建立操作日志与关键业务过程留痕
- 收紧账套删除前置条件

### 4.3 目标三：搭建电子凭证与电子档案底座

- 新增电子凭证记录、校验、去重、解析、入账关联
- 输出归档所需元数据、结构化入账文件和导出清单
- 区分“系统备份”与“归档导出”

### 4.4 目标四：补齐最低输出能力

- 补齐账簿查询
- 补齐报表输出
- 补齐打印、版式导出、标准数据导出接口

## 6. 关键业务规则

### 6.1 账套与适用范围

- 一个账套对应一个委托单位
- 一个委托单位仅允许选择 `enterprise` 或 `npo`
- 不提供政府会计和事业单位模板

### 6.2 凭证控制

- 已记账凭证默认不可逆、不可删除、不可修改关键字段
- 普通用户仅能对未记账凭证执行审核、反审核、删除等允许动作
- 反记账为管理员可分配权限；执行时必须记录理由、审批标记和时间
- 新建记账凭证默认日期沿用上一张凭证日期；当前期间无凭证时默认当月 1 日
- 凭证列表与凭证录入导航统一按凭证字号排序：记账凭证在前，结账凭证在后
- 闭账后允许从凭证管理双击进入只读查看，但不允许修改
- 系统必须具备防止电子会计凭证重复入账的能力
- 末级损益科目必须先配置结转目标，未完整配置或目标失效时不得执行期末损益结转

### 6.3 电子凭证处理

- 原始电子凭证需保留来源、类型、校验状态、解析状态和摘要信息
- 解析出的结构化结果需可追溯到原文件
- 入账后需建立“原始电子凭证 -> 记账凭证 -> 归档导出物”的关联链路

### 6.4 电子档案与备份

- 备份包与档案导出包必须分离
- 档案导出包必须保留元数据、校验信息和导出记录
- 备份恢复必须先校验包完整性

### 6.5 输出能力

- 软件应能输出会计凭证、账簿、报表
- 输出方式至少包含：显示、打印、版式导出、标准数据导出
- 民间非营利组织报表取数应使用显式“科目/现金流量项目 -> 报表行”映射，并兼容父级科目下新增明细科目汇总

## 7. 非功能要求

- 全部核心数据本地存储
- 所有关键业务操作需要可查询日志
- 重要导出与恢复动作要有结果反馈与失败原因
- 文档统一 UTF-8 中文
- 新增模块优先通过 IPC 和服务层扩展，不把复杂规则下沉到 renderer

## 8. 验收标准

### 8.1 文档

- `prds/prd.md`、`prds/PROJECT_SPEC.md`、`prds/合规整改计划.md` 与当前代码状态一致
- 产品定位、范围边界和分阶段任务明确

### 8.2 核心控制

- 普通用户无法反记账已记账凭证
- 具备反记账权限的账号执行反记账时必须填写原因、审批标记并产生日志
- 删除账套时，缺少备份或档案导出记录会被阻止

### 8.3 合规底座

- 能创建电子凭证记录并执行重复入账拦截
- 能生成备份包并进行恢复校验
- 能生成电子会计档案导出包、执行包级校验并保留导出记录

- `npm run typecheck` 通过
- `npm test` 通过
