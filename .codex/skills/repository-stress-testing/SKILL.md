# repository-stress-testing

用于把“压力测试”从一次性拍脑袋操作收敛成可复用的仓库级 workflow。重点不是单次压满，而是证明关键数据链路在高频、长时间和数据累积场景下会收敛，不会在运行一段时间后因重复写入、资源泄漏、快照漂移、后台堆积或恢复放大而系统性崩溃。

## 快速开始

1. 在仓库根目录创建本轮压测工作区：

```powershell
py -3 .codex/skills/repository-stress-testing/scripts/init_stress_workspace.py <slug> --root "<repo-root>"
```

2. 先填写生成出的 `module-matrix.md` 和 `invariants.md`，不要先上来就写压测脚本。
3. 按“基线 -> 高频 burst -> 数据累积 -> 长时间 soak -> 重启/恢复 -> 收尾对账”的顺序执行。
4. 每一阶段都记录实际命令、样本量、观测信号和失败证据；没有执行记录，就不算完成验证。

如果项目已有自己的任务面板、进度文档或上下文快照，先遵守项目规则，再使用本 Skill。

## 工作流

### 1. 建立系统边界

- 先识别真实运行单元：UI、API、主进程、后台任务、数据库、缓存、导入导出、日志、定时器、文件系统、第三方依赖。
- 明确“正常结束态”是什么，不要只定义“开始负载”。
- 查清哪些模块会写库、写文件、发事件、触发异步重试或二次派生计算。
- 对每个模块记录：
  - 入口
  - 主要读写对象
  - 下游触发链
  - 是否可能形成循环
  - 现有保护（幂等、唯一约束、去重、锁、TTL、清理任务）

使用 [`references/stress-matrix-template.md`](./references/stress-matrix-template.md) 作为填写模板。

### 2. 先定义收敛条件，再设计压测

对每一条关键链路，先写清楚以下内容：

- 触发动作是什么。
- 会产生哪些状态变化。
- 哪些变化会再次触发上游或旁路逻辑。
- 多久之内应回到稳定态。
- 稳定态如何判定。

至少给出一类明确的不变量：

- 队列/任务数在停止输入后回落到 0 或稳定平台。
- 同一输入重复执行不会无限新增凭证、记录、文件、日志或任务。
- 派生快照与源数据在固定窗口内可对账。
- 内存、句柄、临时文件、数据库体积、索引大小、日志增长率不应持续单调失控。
- 重启、恢复、重试不会把同一批任务放大成更大的积压。

如果一条链路无法定义收敛条件，默认按高风险处理。

### 3. 设计 5 类测试而不是只做一种

#### 基线测试

- 先跑一次最短正常链路，确认环境、账号、样本数据和关键命令无误。
- 记录基线指标：响应时间、数据库大小、日志文件大小、队列长度、内存/CPU、关键表记录数。

#### 高频 burst

- 在短时间内重复触发高频入口，观察瞬时错误、重复写入和锁竞争。
- 典型入口：保存、导入、审核/记账、报表生成、导出、同步、重试、批量操作。

#### 数据累积 ramp

- 分批扩大数据量，而不是一步到位。
- 每个阶段都做一次对账，确认不是“错误先发生，后续再被更大噪音掩盖”。
- 尤其关注分页、汇总、索引、清理任务、历史快照、缓存淘汰、文件目录扫描等随规模放大的逻辑。

#### 长时间 soak

- 在接近真实节奏的负载下运行较长时间，观察资源是否泄漏、漂移是否累积、错误是否从偶发变成稳定复现。
- 优先让系统自行跑完完整生命周期：创建、处理、清理、归档、重启、恢复。

#### 恢复与 restart

- 在有积压、有中间态、有部分失败的情况下执行重启、恢复、重新导入或任务续跑。
- 目标是验证恢复路径是否收敛，而不是仅验证“能重新启动”。

使用 [`references/failure-patterns.md`](./references/failure-patterns.md) 对照隐藏风险信号。

### 4. 观测信号必须覆盖“资源”和“业务”两侧

至少同步采集两类信号：

- 业务侧：
  - 关键表记录数
  - 重复记录数
  - 快照/汇总对账结果
  - 任务积压量
  - 失败重试次数
  - 导出/导入成功率
- 资源侧：
  - 内存
  - CPU
  - 句柄/文件数
  - 数据库大小与 WAL/日志增长
  - 临时目录体积
  - 错误日志增速

不要只看“测试通过率”。长期稳定性问题往往先出现在资源斜率、重复任务和对账漂移里。

### 5. 发现问题后先分类，再决定是否改代码

先区分以下三类问题：

- 产品缺陷：数据不一致、循环不收敛、恢复放大、资源泄漏、隐性重复写入。
- 观测缺口：系统可能有问题，但当前缺少日志、计数器、探针或对账面。
- 测试缺陷：脚本自身不稳定、数据集不确定、环境波动导致误报。

只有证据足够时才修改产品代码；如果只是观测缺口，优先补最小探针或临时诊断脚本，不要直接重构主逻辑。

### 6. 结果输出格式

每个发现至少写清楚：

- 风险等级：P0 / P1 / P2 / P3
- 现象
- 触发条件
- 何时开始出现（数据量、运行时长、并发量、重启次数）
- 非收敛信号
- 实际证据
- 最小复现路径
- 可能根因
- 临时止血方案
- 建议的修复验证方式

如果项目允许写文档，优先把这些内容落到工作区里的 `findings.md` 与 `soak-log.md`。

## 必守规则

- 不要把 smoke test 误当成稳定性验证。
- 不要只验证“有输入时能成功”，还要验证“停止输入后会收敛”。
- 不要只测业务正确性，还要测资源曲线。
- 不要在缺少基线的情况下直接长跑。
- 不要只跑一次；至少包含一次基线和一次放大量级后的复验。
- 不要因为短时间内没崩就宣布稳定。
- 不要忽略清理、补偿、重试、重启和恢复路径。
