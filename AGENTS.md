# AGENTS.md

全局指令：

- 所有交流统一使用简体中文；如无特殊说明，项目内新增或修改的文案默认使用简体中文。
- 完成一次重要修改后，必须先做 review，再执行验证；验证通过后再向用户交付成果。
- 当前打开的 folder 视为项目级最底部目录；除非用户明确要求，过程中新增文件均放在当前项目目录内。
- 如用户要求打包为单独文件、安装包或其他可交付安装产物，输出目录统一为 `D:\coding\completed\项目名称`，本条优先于“文件放在当前项目目录内”的规则。

- 每次新对话开始前，先读取 `prds/PROJECT_SPEC.md`，确保背景信息一致。
- 如发生重大变化（例如架构、数据模型、权限模型、核心流程变化），必须先更新 `prds/PROJECT_SPEC.md` 再继续开发。
- 每次完成开发任务后，必须在 `prds/开发日志.md` 追加一条中文记录。
- 记录内容包含日期（YYYY-MM-DD）与完成要点（条目形式）。
- 开发完成任务后，必须进行测试，确保功能正常。

产品与范围约束：

- 本项目产品定位固定为“代理记账企业为委托单位记账使用的单机版财务软件”。
- 当前仅支持两类账套：`enterprise` 与 `npo`。
- 明确不支持政府会计、事业单位会计、云端多端同步、完整内置电子档案库。
- 如新增账套模板、报表模板、科目体系或初始化逻辑，不得引入政府或事业单位口径；如需企业科目体系，严格遵循现有规范与技能约束。

合规开发约束：

- 报表、账簿、备份、电子凭证、电子会计档案导出、操作日志属于合规必选能力，不得再按“未来可选功能”处理。
- 已记账凭证默认不可逆。普通用户不得执行 `unbookkeep` 或任何等效反记账能力。
- 如确需保留已记账逆转路径，只允许作为“管理员紧急逆转”实现，且必须同时满足：强制填写原因、强制填写审批标记、写入可查询可导出的操作日志。
- 不得允许通过物理删除、插入式重排、直接修改关键字段等方式绕过已记账不可逆控制。
- 账套备份主流程应以“单账套备份包”为单位，不再默认以整个数据库作为备份目标；历史整库快照仅允许作为 legacy 记录继续校验/删除，不再作为新流程主入口。
- 账套删除前，备份与电子档案导出改为“强提示 + 显式风险确认 + 操作日志留痕”，不再作为硬阻断前置条件；不得静默放行删除。
- 涉及登录、权限变更、账套模板切换、凭证审核/记账/逆转、结账/反结账、备份、档案导出等关键动作时，必须同步检查是否需要补充操作日志。

数据与接口约束：

- 涉及合规模块开发时，优先复用或维护以下模块：`AuditLogModule`、`BackupRecoveryModule`、`ElectronicVoucherModule`、`ArchiveExportModule`、`ReportingOutputModule`。
- 打包版升级、重装或变更安装目录时，禁止将主数据库、用户设置、用户偏好、壁纸和默认日志等用户数据绑定到安装目录；默认应落在稳定的 `userData` 范围内，并对历史错误路径提供兼容迁移。
- 新增或修改 IPC 时，必须同步更新 preload 暴露与类型声明，确保 `src/main`、`src/preload`、`src/renderer` 三层接口一致。
- 涉及数据库结构调整时，除修改建表 SQL 外，还必须考虑旧库兼容迁移与补字段逻辑。
- 与电子凭证相关的改动，应保持“接收 -> 验签/验真 -> 解析 -> 去重 -> 入账/预填 -> 留痕”的链路完整性，不要跳过校验与留痕直接入账。
- 与归档、备份相关的改动，必须明确区分“账套级备份包”“legacy 整库快照”“归档导出文件”，不得混用概念或互相替代。

文档同步约束：

- 如合规整改范围、阶段目标、验收口径发生变化，除更新 `prds/PROJECT_SPEC.md` 外，还应同步更新 `prds/合规整改计划.md`。
- 如新增重要合规模块、表结构、IPC、权限规则，应在文档中明确记录，不允许只改代码不改文档。

测试约束：

- 涉及合规改动时，优先补充或更新以下测试：已记账不可逆、管理员紧急逆转留痕、重复入账拦截、电子凭证状态流转、归档清单生成、备份恢复校验。
- 至少执行 `npm run typecheck` 与 `npm test`；如某项无法执行，必须在结果说明中明确指出原因。
